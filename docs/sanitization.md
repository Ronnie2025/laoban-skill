# 脱敏规则

任何公开发布之前，都必须先做脱敏。

## 必须处理的对象

以下信息要么替换，要么删除：

- 人名
- 公司名
- 部门名
- 产品名
- 客户名
- 项目代号
- 私有 URL
- 本地文件路径
- 精确敏感数字
- 只在某个组织内部成立的黑话和缩写

## 替换策略

如果上下文还需要保留，用稳定占位符：

- `老板A`
- `同事B`
- `客户C`
- `产品X`
- `系统Y`
- `工作区Z`

如果具体身份不重要，直接写成泛化描述：

- `一次内部评审流程`
- `一个面向客户的流程`
- `一次线上问题升级会议`

## 保留功能，删除身份

好的脱敏，不是把内容写空，而是把“方法”留下，把“身份指纹”删掉。

### 不好的写法

- “某位具名老板在评审某个具名产品定价需求时会说……”
- “/absolute/private/path/notes/…”

### 更好的写法

- “当该角色评审定价相关需求时，通常会先追问范围、边界和商业逻辑。”
- “私有工作目录”

## 数字处理

如果数字敏感，就保留相对关系，不保留精确值：

- `17 个客户` -> `一批真实但规模不大的客户`
- `120000` -> `一笔有明确商业意义的金额`

如果数字对方法本身很重要，而且不敏感，可以保留。

## 原话处理

原话是最容易暴露人物身份的部分之一。

公开版建议：

- 优先用转述，不直接引用
- 只保留可泛化的表达习惯
- 删除高度可识别的特殊措辞

## 发布前检查

建议至少做两轮检查：

1. 关键词扫描
   扫人名、公司名、客户名、产品名、用户名、本地路径根目录。
2. 人工复核
   看有没有间接暴露身份的信息，比如非常独特的例子、语气或口号。
